# Phase 8 - Smart Contract Development

This phase adds the Solidity smart contract for storing and verifying certificate hashes. It does not deploy the contract to Ganache and does not connect Django to blockchain yet.

## Goal

Create a simple `CertificateStore` smart contract that can:

- Store a certificate hash.
- Verify a certificate hash.
- Retrieve stored certificate data.

## Files Added

```text
smart_contract/
├── CertificateStore.sol
└── README.md
```

## Contract Overview

The contract stores one record per certificate ID.

Each record contains:

- `certificateHash`: the SHA256 hash generated by the Django app.
- `issuer`: the wallet address that stored the hash.
- `issuedAt`: the blockchain timestamp when the hash was stored.
- `exists`: whether the certificate ID already has a stored record.

## Main Functions

### `storeCertificateHash`

Stores a new certificate hash.

Inputs:

- `certificateId`: certificate ID from Django.
- `certificateHash`: SHA256 hash from Django.

Rules:

- Certificate ID cannot be empty.
- Certificate hash cannot be empty.
- The same certificate ID cannot be stored twice.

### `verifyCertificateHash`

Checks whether a certificate hash matches the stored blockchain hash.

Inputs:

- `certificateId`: certificate ID to verify.
- `certificateHash`: hash to compare.

Returns:

- `true` if the certificate exists and the hash matches.
- `false` if the certificate does not exist or the hash is different.

### `getCertificate`

Returns the stored certificate details for a certificate ID.

Returned values:

- certificate hash
- issuer wallet address
- issued timestamp
- exists status

## Compilation Steps In Remix

1. Open `https://remix.ethereum.org/`.
2. Create a new file named `CertificateStore.sol`.
3. Copy the code from `smart_contract/CertificateStore.sol`.
4. Open the **Solidity Compiler** panel.
5. Select compiler version `0.8.20` or newer.
6. Click **Compile CertificateStore.sol**.
7. Confirm that Remix shows no compiler errors.

## Local Testing Steps In Remix

Use Remix's built-in test environment for simple contract testing during this phase.

1. Open the **Deploy & Run Transactions** panel.
2. Set **Environment** to **Remix VM**.
3. Select `CertificateStore`.
4. Click **Deploy**.
5. Use this sample data with `storeCertificateHash`:

```text
certificateId: CERT-001
certificateHash: 8a7f2c4b1d9e0f1234567890abcdef1234567890abcdef1234567890abcdef12
```

6. Call `verifyCertificateHash` with the same values.
7. Confirm the result is `true`.
8. Call `verifyCertificateHash` with the same certificate ID but a different hash.
9. Confirm the result is `false`.
10. Call `getCertificate` with `CERT-001`.
11. Confirm it returns the stored hash, issuer address, timestamp, and `exists = true`.

## Beginner Explanation

The Django app creates certificate data and hashes it using SHA256. The smart contract stores only that hash, not the full certificate.

This keeps the blockchain record small and avoids putting student details directly on-chain. Later, verification will work by hashing the certificate again and comparing that new hash with the hash stored in this contract.

If both hashes match, the certificate data has not changed. If the hashes do not match, the certificate may have been modified.

## Common Errors And Fixes

### Compiler version error

Use Solidity compiler `0.8.20` or newer in Remix.

### Certificate already exists

The contract blocks duplicate certificate IDs. Use a new certificate ID for each test.

### Empty value error

Both `certificateId` and `certificateHash` are required.

### Verification returns false

Check that the certificate ID and hash are exactly the same as the values stored earlier.
